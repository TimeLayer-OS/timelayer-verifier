# Метод работы агента: Квитанции + Мозг

Это инструкция по **методу работы**, а не по конкретному проекту. Любой агент, который берётся
за ремонт, аудит или любую многошаговую задачу, работает по этой логике. Ссылки на репозиторий,
на мозг-волт и ключи/доступы задаёт владелец в начале сессии. Твоя задача — не «сделать вид,
что готово», а закрыть каждый пункт с доказательством.

Метод даёт: **работа за один проход**, ничего не ломается, агент не путается, любой результат
можно перепроверить постфактум — по коммитам, тестам, sha и журналу.

---

## Почему это работает (наблюдение, проверено на практике)

Два простых правила снимают две главные болезни агентов — выдумывание «готово» без проверки и
потерю контекста между шагами. Эффект на практике:

- **Один проход вместо многих.** Хорошо поставленная задача проходит как **один аудит →
  один ремонт**, без циклов «переделай». Агент не возвращается, потому что каждый шаг закрыт
  доказательством, а не обещанием.
- **Спека через мозг — кратно быстрее и дешевле.** Если сначала загнать спецификацию и план в
  мозг (внешнее состояние), а потом работать из него, задача делается **в разы быстрее и с
  заметно меньшим расходом токенов**: агент не перечитывает всё заново и не держит контекст в
  одном раздутом окне — он подхватывает актуальное состояние и продолжает с места.
- **Качество выше.** Проверка по коду и квитанции, а не по словам, убирает тихие ошибки и
  регрессии до того, как они накопятся.

Коротко: **думай мозгом (внешним состоянием), закрывай квитанцией (доказательством), проверяй
по коду (не по словам).**

---

## 1. Два столпа

**КВИТАНЦИЯ** — ни один пункт не считается сделанным без доказательства. Доказательство — это
не слова «я починил», а проверяемый факт: хэш коммита, результат теста (red → green), вывод
команды, внешняя проверка. Нет квитанции → пункт открыт.

**МОЗГ** (внешний волт состояния) — планы, журнал действий и решения живут не в памяти одного
прохода, а в постоянном хранилище (репозиторий-волт). Между сессиями состояние не теряется,
поэтому не нужно держать всё в голове и не возникает путаницы.

---

## 2. Дисциплина тикета

Каждая единица работы = ТИКЕТ. У тикета есть статус и обязательная квитанция при закрытии.

Статусы: **open** (дыра описана, не тронута) · **в работе** · **closed** (сделано, с квитанцией —
иначе закрывать нельзя) · **отложено** (с явной причиной: окно обслуживания / условие / зависимость).

Тело тикета всегда содержит:
1. **Корень** — что именно не так, с точным адресом (`файл:строка`, а не «где-то в модуле»).
2. **Действие** — что конкретно сделать.
3. **Критерий приёмки** — как понять, что дыра закрыта (какой тест/проверка это докажет).
4. **Квитанция** — заполняется в момент закрытия (доказательство).

Пример закрытого тикета:
```
## A2 — [HIGH] Клиент без таймаута морозит цикл
open → closed
Корень: src/info_client.rs:23 — Client::new() без таймаута.
Действие: builder с timeout(8s)/connect_timeout(5s).
Критерий приёмки: тест с mock-сервером, молчащим >8s → ошибка таймаута, не вис.
Квитанция:
- Правка info_client.rs:23. Тест client_times_out red→green. 153 passed; 0 failed.
- Коммит <хэш>. Развёрнуто, sha бинаря изменился с <старый> на <новый>.
```

---

## 3. Что считается квитанцией (по типам работы)

| Тип работы | Валидная квитанция |
|---|---|
| Правка кода | хэш коммита + тест red→green + `N passed; 0 failed` |
| Деплой | sha256 бинаря **до и после** + подтверждение, что развёрнуто на всех нужных хостах |
| Инфра/конфиг | вывод команды ДО и ПОСЛЕ (напр. `iptables -L -n`, коннект снаружи → отказ) |
| Проверка чужой работы | подтверждение по коду/бинарю (grep правки, сверка sha), **не** по чужому отчёту |
| «Ложная тревога» | доказательство, почему находка неверна (разбор логики/математики), затем откат |
| Отложено | причина + условие возобновления + куда записан план догона |

Правило: квитанция должна быть **воспроизводимой другим**. Если её нельзя перепроверить — это
не квитанция.

## 4. Работа с GitHub (код = источник истины)

1. Владелец даёт ссылку на репозиторий и доступ. Работаешь в нём, а не по пересказам.
2. Каждая правка = отдельный коммит с понятным сообщением: **что** починено и **почему**.
   Хэш коммита — это часть квитанции.
3. Перед «сделано» прогоняешь тесты и вставляешь их результат в квитанцию.
4. Сборка релиза — фиксируешь sha256 артефакта. Деплой — сверяешь, что на всех хостах
   развёрнут **один и тот же** sha.
5. **Проверка = по коду, не по отчёту.** Открываешь файл, делаешь grep правки, сверяешь sha
   развёрнутого бинаря. Отчёт может врать или отставать — код и бинарь не врут. Неизменный sha
   бинаря = слой не трогали, что бы ни говорил отчёт.

---

## 5. Работа с МОЗГОМ (волт состояния)

1. Владелец даёт ссылку на мозг-волт и доступ. Это отдельное постоянное хранилище планов и
   журнала (не код проекта).
2. Структура, которую держим в мозге:
   - **Планы** — файлы плана ремонта/аудита с тикетами (см. §2). Каждый раунд — свой план,
     старые не затираем, а ссылаемся на них.
   - **Журнал (append-only)** — лента: что сделано, когда, с какой квитанцией. Только дописываем.
3. **Начало сессии: сначала читаешь мозг** — актуальный план и последние записи журнала, чтобы
   продолжить с места, а не начать заново. (Здесь и берётся выигрыш в скорости и токенах.)
4. Конец действия: пишешь квитанцию и в тикет плана, и в журнал. Действие без записи в мозг =
   потерянное действие.
5. Ссылки между записями — либерально (план ↔ журнал ↔ находки), чтобы состояние было связным.

---

## 6. Цикл одного прохода

```
1. Прочитать мозг: актуальный план + журнал (где остановились).
2. Взять следующий open-тикет по приоритету (критичное → высокое → среднее → низкое).
3. Найти корень в коде (файл:строка), не по памяти.
4. Сделать правку. Написать тест на исходный отказ (red → green).
5. Собрать / прогнать тесты. Получить доказательство.
6. Закрыть тикет с квитанцией. Записать в журнал.
7. Если деплой — сверить sha на всех хостах.
8. Перейти к следующему тикету. Ничего не пропускать молча:
   пропуск/ограничение = отдельная запись, а не тишина.
```

## 7. Приоритизация находок

По убыванию денежного/безопасностного риска:
1. **CRITICAL** — активная потеря денег / раскрытие ключа / потеря данных.
2. **HIGH** — латентная дыра, выстреливающая при сбое (обрыв сети, рестарт, переполнение).
3. **MEDIUM** — логика «есть, но не работает» (мёртвые гейты, тихие деградации).
4. **LOW** — шлифовка, устойчивость к битым данным, гигиена.

Критичное закрывается первым, шлифовка — последней.

---

## 8. Железные правила (нарушать нельзя)

- **Нет квитанции — нет «сделано».** Слова не закрывают тикет.
- **Проверяй по коду и бинарю, а не по отчёту.** Отчёт — гипотеза, код — факт.
- **Тест на исходный отказ обязателен** для нетривиальной правки (red → green).
- **Деньги/безопасность:** по умолчанию fail-closed. При сомнении — останавливайся и спрашивай,
  а не «примерно чини».
- **Не ломай соседнее.** Регрессия — тоже открытый тикет; лови её проверкой.
- **Тишина запрещена.** Любой пропуск, ограничение или отложенное — явная запись в мозг.
- **Ничего необратимого без подтверждения** (деплой на живой аккаунт, удаление, рестарт лидера) —
  сначала докажи безопасность, потом делай, в окне.

---

## 9. Что даёт метод

- Работа за один проход: не приходится возвращаться и переделывать.
- Ничего не ломается: каждая правка подтверждена тестом, регрессии ловятся проверкой.
- Агент не путается: состояние в мозге, а не в голове одного контекста.
- Меньше токенов, выше скорость: спека и план живут в мозге, а не перечитываются каждый раз.
- Любой результат перепроверяем: по коммитам, тестам, sha и журналу — постфактум и другим.

Применимо к любой агентной / мультиагентной системе, не только к этому проекту.
